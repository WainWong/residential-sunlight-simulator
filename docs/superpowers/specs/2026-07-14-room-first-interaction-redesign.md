# 房间优先的建模与采光交互重设计

日期：2026-07-14

状态：设计已确认，待实现计划

## 1. 背景

一次新用户观察暴露出当前界面依赖大量未被表达的内部概念：

1. 用户不理解“建筑进深”。
2. 用户能看出旋转数值会改变建筑，但无法把角度与最终朝向对应起来。
3. 用户误以为 X/Y 坐标是建筑本身的参数。
4. 用户不理解“添加观察区”。
5. 用户不理解“画区 / 擦除”，画错后首先尝试 `Ctrl+Z`。
6. 用户完成绘制后不知道还要从“编辑”切到“展示”。
7. 用户进入展示后不知道还要再次“进入观察区”。
8. 用户不知道观察区贴外墙会自动产生洞口。

这些问题不能靠教程、弹窗或更多解释文案解决。当前“观察区”同时承担房间、分析目标和自动洞口三个概念；编辑/展示、选择分析对象和相机进入室内也被拆成多个隐藏状态。界面暴露了存储结构和模式，而不是用户任务。

本设计将产品改为现实世界的对象顺序：

> 建筑 -> 房间 -> 墙上开口 -> 查看直射日光

允许在角落保留低干扰的简短步骤说明，但主要依靠对象、直接操作和上下文动作建立可发现性。

## 2. 已确认的产品边界

- 本版以一个完整版本交付，不让“房间”和“观察区”在用户界面中并存。
- 内部实施可以拆成多个里程碑，但最终一起切换数据模型、术语和主流程。
- 桌面和平板支持完整建模；手机只支持浏览、房间切换、日期时间和采光结果。
- 仍只计算直射日光，不计算天空漫射、间接反射、多次反射或玻璃折射。
- Three.js 环境补光仅用于看清模型，不属于采光结果。
- 家具本版不实现，只在房间数据中预留归属结构。
- 用户不需要绘制完整楼层，只绘制目标房间和直射路径所需的相邻空间。
- 建筑旋转连续，不做方向吸附。
- 建筑编辑不提供单独“俯视”按钮。

## 3. 设计目标

1. 用户通过拖动模型而不是解释坐标来完成建筑位置、尺寸和方向调整。
2. 用户创建的是房间，不再接触“观察区、画区、擦除、进入观察区”。
3. 同一创建会话中的多个相连矩形组成一个房间；新建会话创建另一个房间。
4. 墙由房间和建筑几何关系派生，开口显式添加，不再由房间贴墙自动产生。
5. 点击墙即可添加窗、落地窗、门洞或半墙开口，并区分玻璃与完全开放。
6. “搭建场景 / 查看采光”继续分阶段，以保持编辑性能和信息聚焦。
7. 不依赖用户发现顶栏切换；完成房间和开口后在右侧原地出现“查看采光”主按钮。
8. 从房间面板进入查看采光时直接进入该房间；点击建筑返回室外。
9. `Ctrl+Z`、撤销和重做覆盖所有几何编辑，一次指针手势只产生一条历史记录。

## 4. 非目标

- 不做漫反射、天空光、全局光照、路径追踪或绝对照度值。
- 不承诺法规级日照或采光合规结论。
- 不做完整 CAD 墙线编辑器。
- 不要求用户绘制整层户型。
- 不实现床、桌、柜等家具库、碰撞和摆放交互。
- 不实现真实门扇及开关角度；本版只提供允许光线通过的“门洞”。
- 不实现凸出建筑轮廓的悬挑阳台。开放阳台需求由“半墙开口”覆盖。
- 不在手机端提供复杂几何编辑。

## 5. 整体信息架构

桌面端保持三栏工作台：

- 左侧：建筑和房间对象树。
- 中央：Three.js 场景与直接操作器。
- 右侧：当前选中对象的上下文面板。
- 底部：只在“查看采光”阶段出现的日期和时间轴。

顶栏保留“搭建场景 / 查看采光”作为熟练用户的快捷切换和当前状态显示，但不承担主流程发现任务。

右侧上下文主动作按状态变化：

| 当前状态 | 主动作 |
|---|---|
| 建筑没有房间 | `添加房间` |
| 当前房间直接临外墙但没有开口 | `添加窗或开口` |
| 当前房间不临外墙且没有相邻空间 | `添加相邻房间` |
| 已有相邻空间但尚未连通 | `添加门洞或室内窗` |
| 当前房间具备可分析几何 | `查看采光 ->` |

顶层阶段仍保留的原因：

- 搭建阶段不需要日期、时间和真实太阳光。
- 搭建阶段停止采光分析和太阳渲染，保持编辑响应速度。
- 查看阶段锁定几何编辑，专注太阳、阴影、房间结果和遮挡来源。

## 6. 建筑直接操作

### 6.1 术语与表单

- 用户界面的“建筑进深”统一改为“建筑宽度”。
- 默认界面移除 X/Y 或 X/Z 坐标输入。
- 右侧仍保留名称、长度、宽度、楼层、层高和旋转角度，作为精确修正手段。
- 内部字段 `depth` 可在迁移期保留，但对外统一显示“宽度”。

### 6.2 移动

- 拖动建筑本体移动建筑。
- 指针按下后暂时锁定相机操作；松手后恢复相机旋转。
- 拖动过程中显示与最近建筑外墙的净距。
- 不显示相对拖动起点的东/西/南/北位移。
- 用户以大致摆放为主，不要求站点原点或坐标系知识。

### 6.3 尺寸

- 选中建筑后在四个边中点显示尺寸手柄。
- 长边中点和短边中点分别修改宽度和长度。
- 模型旁实时显示长度和宽度尺寸线。
- L 形和回字形使用轮廓上的分段尺寸，不再用单一“进深”解释全部几何。

### 6.4 旋转

- 选中建筑后在地面显示旋转环。
- 用户可从圆环任意位置开始拖动。
- 拖动时从建筑中心向指针落在地面的点伸出箭头，建筑实时跟随箭头旋转。
- 箭头旁显示方向和角度，例如 `东南 137°`。
- 松手后长箭头收回，只保留圆环上的当前方向手柄。
- 旋转完全连续，不吸附正南、东南等方向。

### 6.5 历史记录

- 拖动建筑、调整尺寸或旋转，从 pointer down 到 pointer up 只提交一条命令。
- 顶部工具区提供撤销和重做图标。
- 键盘支持 `Ctrl+Z`、`Ctrl+Shift+Z`，并兼容平台对应的 Meta 快捷键。

## 7. 房间创建与编辑

### 7.1 入口与楼层

- 左树中每栋建筑提供 `+`，右侧建筑面板提供“添加房间”。
- 点击后先确认楼层，默认沿用最近编辑楼层或第 1 层。
- 系统自动聚焦该楼层并转到接近俯视的角度；用户随后仍可自由旋转视角。
- 建筑编辑阶段不单独显示“俯视”按钮。

### 7.2 创建会话

- 点击“添加房间”只创建临时会话，不立即写入正式房间数组。
- 用户直接拖出第一个矩形，不需要选择“画区”。
- 完成前继续拖出的相连矩形自动并入当前房间。
- 新矩形必须与当前房间相连。
- 与其他房间重叠、超出建筑内部或与当前房间断开时显示红色无效预览，不提交数据。
- 每个矩形拖拽都是独立历史步骤，可以逐块撤销。
- 右侧明确显示“取消 / 完成房间”。

### 7.3 完成后编辑

- 完成后房间进入普通选中状态。
- 用户可拖动房间轮廓边、角和矩形块继续调整，不使用“擦除模式”。
- 局部删除通过选择组成块后按 Delete 或使用删除动作完成。
- 房间类型可选：客厅、卧室、书房、厨房、阳台、其他。
- 未选择类型时自动命名为“房间 N”；选择类型后可自动命名为“卧室 N”等，名称仍可手动修改。

### 7.4 房间身份

- 同一创建/编辑会话中的相连矩形属于同一 `roomId`。
- 完成后再次“添加房间”产生新的 `roomId`。
- 几何接触永远不会自动把两个不同房间合并。
- 两个房间相邻时，共享边界派生为共用墙。

## 8. 墙与开口

### 8.1 墙的可见交互

- 用户不需要先理解“内墙 / 外墙”。
- 墙面悬停时描边，点击后进入墙面上下文。
- 标题显示被点击墙面的朝向，例如 `墙面朝向：西南 225°`。
- 朝向来自被点击墙面的法线；共用墙按用户点击的那一侧计算。
- 选中墙后在点击位置显示 `+`，开口初始中心位于该位置。
- 相机保持当前视角，不自动跳转。
- 上下文提供“正视墙面”相机快捷动作；它不是新的编辑模式。

### 8.2 开口预设

底层统一为“墙上开口”，预设只提供默认几何和填充：

| 预设 | 默认几何 | 默认填充 |
|---|---|---|
| 窗 | 有窗台、中等宽高 | 玻璃 |
| 落地窗 | 接近地面、较大宽高 | 玻璃 |
| 门洞 | 从地面开始、门宽和门高 | 完全开放 |
| 半墙开口 | 保留约 1.1 米矮墙，上方可铺满墙段 | 完全开放 |

“半墙开口”覆盖当前开放式阳台需求，不创建独立阳台平台。

### 8.3 开口编辑

- 左右手柄修改宽度。
- 上下手柄分别修改底部和顶部高度。
- 右侧同步显示宽度、底部高度和顶部高度。
- 开口可以从预设继续修改为自定义尺寸。
- 填充属性独立为 `glass` 或 `open`。
- 玻璃和完全开放在三维画面中必须明确区分。
- 当前直射面积算法对两者都视为可通过；数据中预留未来玻璃透射率，但本版不输出光强或照度。

### 8.4 空间连接

- 房间边界与建筑边界重合的墙连接房间与室外。
- 两个房间共享边界的墙连接两个房间。
- 仅一侧有房间且另一侧未建模的边界视为封闭墙。
- 门洞、室内窗和开放口可以连接两个已建模空间。
- 太阳射线只有连续穿过路径上的所有开口才到达目标房间。

## 9. 查看采光与室内外导航

### 9.1 进入

- 完成房间和开口后，右侧原地出现“查看采光 ->”主按钮。
- 点击该按钮同时完成：切换到查看阶段、设置当前分析房间、相机进入该房间、显示结果面板和时间轴。
- 不再要求用户先切顶部阶段，再点击“进入观察区”。

### 9.2 查看阶段对象行为

- 点击房间：切换当前分析对象、相机进入该房间、右侧显示该房间结果。
- 点击建筑：返回该建筑室外视角，但保留当前分析房间的结果。
- 点击面包屑“室外场景”或按 Esc：返回室外。
- 右上角始终提供“返回搭建场景”，不要求用户寻找顶栏开关。
- 面包屑显示当前位置，例如 `室外场景 / 住宅 1 / 客厅`。

### 9.3 结果语言

结果明确限定为直射日光：

- 当前是否有直射日光。
- 当前直射面积比例。
- 全天直射时段与累计时长。
- 太阳高度角和方位角。
- 室外视角中的遮挡建筑提示。

不使用“整体采光亮度”或“室内照度”等当前算法无法支持的措辞。

## 10. 响应式设计

### 10.1 桌面

- 左树、中央场景和右侧面板同时可见。
- 所有直接操作器和快捷键完整可用。

### 10.2 平板

- 保留完整建模能力。
- 左右面板改为可收起抽屉或底部 sheet。
- 旋转环、尺寸手柄、墙和开口使用更大的不可见命中区域，视觉尺寸保持克制。
- 单指对象操作时锁定相机；空白区域手势控制相机。

### 10.3 手机

- 支持项目打开、建筑和房间浏览、室内外切换、日期时间和结果。
- 隐藏建筑、房间和开口几何编辑入口。
- 用户尝试进入编辑时提示使用桌面或平板继续，不展示残缺的编辑器。

## 11. 数据模型

项目 schema 升级到版本 2。建议核心结构如下：

```js
building = {
  ...,
  rooms: [{
    id,
    floor,
    name,
    type: null | 'living' | 'bedroom' | 'study' | 'kitchen' | 'balcony' | 'other',
    rects,
    objects: []
  }],
  openings: [{
    id,
    floor,
    connectedRoomIds,
    wallAnchor,
    preset: 'window' | 'floorWindow' | 'doorway' | 'parapet' | 'custom',
    bounds: { centerU, width, bottom, top },
    fill: 'glass' | 'open',
    transmittance: null
  }]
}
```

视图状态建议统一表达当前任务和选择：

```js
view = {
  phase: 'build' | 'sunlight',
  selection: null | { kind: 'building' | 'room' | 'wall' | 'opening', id },
  roomEditing: null | { mode, buildingId, roomId, floor, rects, type, name },
  interiorRoomId: null | roomId
}
```

`simulation.activeRoomId` 替代 `activeAreaId`。

## 12. 派生墙与开口锚定

- 墙不作为用户需要维护的长期对象存储。
- 域层根据建筑轮廓、楼层和房间 rect union 派生墙段及相邻空间。
- 开口存储墙面局部锚点、楼层和连接房间 ID，不只依赖易变化的数组索引。
- 房间修改后重新派生墙，并尝试把开口重投影到原墙面关系。
- 墙仍存在但长度变化时，开口保持相对位置并限制在墙内。
- 墙缩短到无法容纳开口、或墙完全消失时，开口进入 invalid 状态。
- invalid 开口标红并阻止完成房间修改；用户必须移动、缩小或删除。
- 系统不得静默删除开口。

## 13. 数据流与性能

直接操作采用两阶段数据流：

```text
pointer gesture
  -> 临时预览状态
  -> 轻量 Three.js 预览网格
  -> pointer up / 完成
  -> 一条 Store Command
  -> 派生房间、墙、开口
  -> 重建正式几何
  -> 更新直射分析输入
```

- 建筑移动、尺寸和旋转拖动期间不运行 CSG。
- 松手后才重建正式建筑、房间和开口几何。
- 房间创建会话使用临时轮廓和墙预览，点击完成后写入项目。
- 搭建阶段不运行太阳和采光分析。
- 查看阶段锁定几何编辑，时间变化只更新太阳、阴影和直射结果。
- 全日计算继续在 Worker 中运行，并保持 latest-wins 时序。
- 当前时刻结果可以同步快速更新；全日结果失败不阻塞当前时刻查看。

## 14. 组件边界

- `features/buildings`：建筑表单和直接操作状态协调。
- `features/rooms`：房间创建/编辑会话和右侧面板。
- `features/openings`：墙选择、开口预设和尺寸编辑。
- `domain/rooms`：相连矩形、房间验证和 rect union。
- `domain/walls`：按楼层派生墙段、墙面朝向和空间邻接。
- `domain/openings`：开口锚定、重投影和有效性检查。
- `domain/simulation`：把所有相关墙和开口转换为 obstacle/portal，复用直射判定。
- `scene/gizmos`：建筑旋转环、尺寸手柄、房间轮廓和开口手柄。
- `scene/navigation`：楼层聚焦、进入房间、返回室外和相机遮挡处理。
- `features/shell`：对象树、上下文主动作和桌面/平板/手机壳层。

场景层不直接修改项目数据；所有正式变更通过 command store 提交。

## 15. 迁移

版本 1 项目迁移到版本 2：

- `observationAreas` 逐个转换为 `rooms`。
- 原 area 的 `floor`、`rects` 和稳定 ID 保留；旧名称缺失时派生“房间 N”。
- 旧 area 与外墙接触产生的自动 aperture 转换为显式 `custom + open` 开口，保持旧项目直射路径。
- 旧 `window` 转换为 `window + glass`。
- 旧 `floorWindow` 转换为 `floorWindow + glass`。
- 旧 `balcony` 开口转换为 `parapet + open`，并保留原宽高。
- `activeAreaId` 转换为 `activeRoomId`。
- `areaEditing`、`editorMode` 和 `interior.areaId` 迁移到新的 selection/roomEditing/interiorRoomId 状态。
- 无法建立有效墙锚点的旧开口保留为 invalid，并在首次编辑时要求修复；导入过程不静默丢数据。

## 16. 错误与空状态

- 无建筑：中央显示空状态，主动作“添加建筑”。
- 建筑无房间：右侧主动作“添加房间”。
- 房间无室外连通路径：根据几何显示“添加窗或开口”“添加相邻房间”或“添加门洞或室内窗”；同时允许主动查看无直射结果。
- 房间越界、重叠或断开：红色预览，不提交。
- 开口越界、重叠或无足够墙面：红色预览，不提交。
- 修改房间导致开口 invalid：阻止完成并列出需要处理的开口。
- Worker 失败：当前时刻直射结果仍可用；全天结果显示“计算失败，可重试”。
- WebGL 不可用：保留项目数据和只读列表，明确说明三维场景不可用。
- 导入迁移失败：保留当前项目，不做部分覆盖，并报告具体对象。

## 17. 测试策略

### 17.1 单元测试

- 相连矩形并集、断开和重叠校验。
- 建筑边界墙、房间共用墙和未建模边界墙派生。
- 墙面朝向与 `西南 225°` 等读数。
- 开口锚点、墙变化重投影和 invalid 状态。
- 玻璃/开放预设与迁移。

## 18. 新用户验收标准

不看教程，只允许角落保留低干扰的简短步骤说明，新用户应能：

1. 调整建筑长度、宽度、位置和方向，不接触“进深”和 X/Y。
2. 添加一个 L 形房间，画错后自然使用 `Ctrl+Z` 或撤销按钮。
3. 选中墙并添加窗或半墙开口。
4. 完成后从右侧“查看采光”直接进入房间。
5. 点击建筑返回室外，判断是哪栋楼造成遮挡。
6. 全程不出现“观察区、画区、擦除、进入观察区”等内部术语。

## 19. 交付策略

作为一个用户可见版本交付，但按以下内部顺序开发和验证：

1. schema v2、房间/墙/开口纯域模型和迁移。
2. 建筑直接操作、统一历史记录和预览提交机制。
3. 房间创建/编辑与对象树。
4. 墙选择、开口编辑和正式 CSG 几何。
5. 多空间直射适配和采光导航。
6. 平板完整编辑、手机只读和全流程回归。

在所有阶段完成前不向用户暴露混合的“观察区 + 房间”界面。

## 20. 已确认决策摘要

- 采用房间优先、墙体派生方案。
- 同一版本完成建筑、房间、墙、开口和采光导航重做。
- 顶栏切换保留，但右侧上下文主按钮承担流程发现。
- 建筑旋转使用连续圆环和指针箭头，不吸附。
- 移除建筑编辑“俯视”按钮。
- 房间创建时自动聚焦楼层。
- 墙面不自动正视，保留可选“正视墙面”。
- 墙面读数使用方向加角度。
- 门只做门洞；开放阳台由半墙开口覆盖。
- 区分玻璃与完全开放，但本版不计算玻璃透射光强。
- 查看采光从当前房间直接进入，建筑节点返回室外。
- 漫反射和家具留待后续版本。

当前设计无未决产品问题；下一步是在用户复核本文后编写详细实现计划。
